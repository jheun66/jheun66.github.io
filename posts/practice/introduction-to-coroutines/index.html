개요

이 글은 안드로이드 개발자 사이트의 내용을 따라해보면서 작성하였습니다.

Multithreading and concurrency

쓰레드는 프로그램의 경계안에서 스케줄되고 실행되는 가장 작은 단위

fun main() {
    val thread = Thread {
        println("${Thread.currentThread()} has run.")
    }
    thread.start()
}
// 실행 결과
// Thread[Thread-0,5,main] has run.

람다함수로 간단한 쓰레드, 쓰레드는 start() 함수가 호출되어야 실행되고 currentThread()는 현재 실행되는 Thread instance를 반환, String 표현으로 쓰레드 이름, 우선순위, group을 알려줌

Challenges with threads

쓰레드는 여러 작업을 동시에 실행하는 가장 쉬운 방법이지만 문제가 없는 것은 아님. 몇가지 어려움이 있음.

Threads require a lot of resources

쓰레드를 만들고 변경하고 관리하는 것은 동시간에 관리해야하는 쓰레드의 제한에 의해 많은 시스템 리소스와 시간을 소모함. (context switching)

앱에서는 특히 UI를 책임지는 하나의 명시적인 쓰레드가 있음. 이를 메인 쓰레드, UI 쓰레드라 함. 따라서 실행을 완료하는데 오랜 시간이 걸리는 작업이 있는데 그전까지 다른 작업을 막으면 앱의 반응이 없게됨.

안드로이드 OS에서는 유저의 반응을 받기위해 여러차례 시도를 함. 현재 휴대기기에서는 1초에 60~120회 정도 UI를 업데이트함.

Race conditions and unpredictable behavior

쓰레드는 하나의 자원을 두고 경쟁할 수 있어서 예측하지 못한 결과를 가져올 수 있음

fun main() {
   var count = 0
   for (i in 1..50) {
       Thread {
           count += 1
           println("Thread: $i count: $count")
       }.start()
   }
}

count라는 변수를 두고 race condition이 발생하는 예

Coroutines in Kotlin

안드로이드에는 백그라운드 작업을 위해 직접 쓰레드를 생성하고 사용할 수 있지만 코틀린에서는 동시성을 관리하는데 더 유연하고 쉬운 방법인 Coroutine을 제공한다.

단순히 쓰레드로 작업하는거 이상의 추상화 레벨을 제공함. 이중 한가지 특징은 상태를 저장하는 능력으로 정지하거나 재실행할 수 있음.

continuations(연속)으로 표현되는 상태는 코드의 일부가 컨트롤을 넘기거나 다시 실행되기전 다른 코루틴이 완료될 떼까지 기다리는 신호를 보내도록 할 수 있음.

연속뿐만 아니라 CoroutineScope내의 lifecycle을 가지는 취소가능한 작업 단위 Job내의 작업을 포함함.

CoroutineScope는 하위 요소들에 재귀적으로 취소와 다른 규칙들을 적용하는 context.

Dispatcher는 코루틴이 실행에 사용할 지원스레드를 관리하여 개발자들이 언제, 어디서 새로운 쓰레드를 사용해야하는지에 대한 책임을 지워줌.

중요 단어 설명
Job launch() 함수에서 생성된 작업처럼 취소 가능한 작업 단위
CoroutineScope launch()와 async() 같이 새로운 코루틴을 생성하기 위해 사용되는 함수는 CoroutineScope를 확장함
Dispatcher 코루틴이 사용할 쓰레드를 결정. 메인 디스패처는 항상 메인 쓰레드에서 코루틴 실행. 다른 Default, IO, Unconfined 디스패처는 다른 쓰레드에서 사용.

Dispatchers는 코루틴이 좋은 성능을 낼 수 있도록하는 방법 중 하나이지만 새 쓰레드를 초기화하는 성능 비용을 피해야함.

import kotlinx.coroutines.*

fun main() {
    repeat(3) {
        GlobalScope.launch {
            println("Hi from ${Thread.currentThread()}")
        }
    }
}

GlobalScope 앱이 실행되는 한 내부의 어떤 코루틴도 실행되도록 허용. 메인쓰레드에 대한 염려가 있어 예시 코드밖에서는 사용하는 것을 추천하지 않음. 앱에서 코루틴을 사용할 때는 다른 스코프를 사용하기

launch() 함수는 취소 가능한 Job 객체로 래핑된 코드로 부터 코루틴을 생성함. launch()는 반환값이 코루틴의 경계밖에서 필요하지않을 때 사용함.

코루틴의 다음 중요한 개념을 이해하기 위해서 launch의 전체 문법을 확인해보면 다음과 같음

fun CoroutineScope.launch {
    context: CoroutineContext = EmptyCoroutineContext,
    start: CoroutineStart = CoroutineStart.DEFAULT,
    block: suspend CoroutineScope.() -> Unit
}

이 코드 블록을 suspend 키워드로 표현함. Suspend는 코드 블록이나 함수가 정지되거나 재개될 수 있음을 나타냄.

A word about runBlocking

import kotlinx.coroutines.*
import java.time.LocalDateTime
import java.time.format.DateTimeFormatter

val formatter = DateTimeFormatter.ISO_LOCAL_TIME
val time = { formatter.format(LocalDateTime.now()) }

suspend fun getValue(): Double {
    println("entering getValue() at ${time()}")
    delay(3000)
    println("leaving getValue() at ${time()}")
    return Math.random()
}

fun main() {
    runBlocking {
        val num1 = getValue()
        val num2 = getValue()
        println("result of num1 + num2 is ${num1 + num2}")
    }
}

다음 예제는 runBlocking, 이름에서도 알 수 있듯이 새 코루틴을 시작하고 현재 쓰레드를 끝날때까지 블락함. 주로 메인 함수와 테스트에서 블락된 코드와 블락되지않은 코드를 연결하는데 사용함. 일반적으로 안드로이드 코드에서는 사용하지 않음.

getValue()는 3초뒤 랜덤 숫자를 반환하는 함수. DateTimeFormatter를 사용해 적절한 들어간 시간, 나온 시간을 보여주고 있음.

entering getValue() at 06:44:53.804
leaving getValue() at 06:44:56.831
entering getValue() at 06:44:56.832
leaving getValue() at 06:44:59.832
result of num1 + num2 is 0.79934378282777

num1과 num2에 사용된 getValue는 독립적이고 정지하는데 코루틴이 필요하지 않음. 이때 async()와 await()를 사용하여 비동기를 구현할 수 있음

fun main() {
    runBlocking {
        val num1 = async { getValue() }
        val num2 = async { getValue() }
        println("result of num1 + num2 is ${num1.await() + num2.await()}")
    }
}

함수 정의는 다음과 같음

Fun CoroutineScope.async() {
    context: CoroutineContext = EmptyCoroutineContext,
    start: CoroutineStart = CoroutineStart.DEFAULT,
    block: suspend CoroutineScope.() -> T
}: Deferred<T>

async() 함수는 Deferred 타입을 반환. Deferred는 미래의 값을 참조하여 가지고 있을 수 있는 취소 가능한 Job. 이를 사용하면 값을 반환받은 것처럼 계속 다음 코드를 호출 가능. 비동기 작업은 언제 값이 리턴되는지 알 수 없으므로 Deferred은 placeholder로써 사용됨. 대신 이후에 값이 리턴되는 것이 보장되어야하고 await()를 사용하여 값이 리턴되는 것을 기다릴 수 있음.

entering getValue() at 06:58:50.172
entering getValue() at 06:58:50.181
leaving getValue() at 06:58:53.18
leaving getValue() at 06:58:53.182
result of num1 + num2 is 1.3197235861568224

When to mark functions as suspend

getValue()에 suspend 키워드가 들어있는 것을 확인할 수 있음. 이는 suspend 함수이기도한 delay()를 호출하기 때문. 함수가 다른 suspend 함수를 호출할 때 이 함수 역시 suspend 함수가 된다. 메인 함수에 suspend가 붙지 않은 것은 getValue()를 직접 호출하는 것이 아닌 runBlocking()을 통해서 호출되기 때문.
