Learn about App Architecture

가장 일반적인 아키텍쳐 원칙은 seperation of concerns와 모델로과 UI의 분리

seperation of concerns

클래스마다 별개의 concern을 나누어 가져야함, 단일 책임의 원칙

Drive UI from a model

쉽게 말하면 model은 데이타로 뷰와는 독립적으로 존재햐아함을 의미

이 챕터에서는 안드로이드 아키텍쳐 중 UI controller(Activity/Fragment)와 ViewModel에 대해 살펴봄

UI controller

화면에 그리는 뷰들을 컨트롤, 유저의 이벤트를 받고 상호작용하는 역활을 함

앱의 데이터나 데이터를 결정하는 로직은 UI controller 클래스 내에 있으면 안됌, ViewModel에 포함해야함

유저의 상호작용이나 low memory 같은 시스템 상태에 따라 파괴될 수 있음

ViewModel

view에 보여줄 앱의 데이터 모델

액티비티나 프레그먼트가 파괴되거나 재생성될때 파괴되지 않음

Fragment에 ViewModel 추가

private val viewModel: GameViewModel by viewModels()

초기화 할때 by viewModels()을 사용하여 초기화를 하는데 이는 코틀린의 프로퍼티 대리자라는 기능

Kotlin property delegate

// Syntax for property delegation
var <property-name> : <property-type> by <delegate-class>()

코틀린에서 var 프로퍼티는 기본적으로 getter와 setter 함수가 자동으로 생성되고 값을 할당하거나 값을 읽을 때 자동으로 호출

그러나 val 프로퍼티는 getter 함수만 자동으로 생성이 되고 값을 읽을때만 호출

property delegation은 이러한 getter와 setter에 대한 책임을 다른 클래스에 이양하는 것

delegate class가 대신 getter와 setter 함수와 기능을 제공하도록 함

예를들어 프로퍼티를 가지고 있는 액티비티나 프래그먼트 클래스가 configuration 변화에 의해 삭제되고 재생성되더라도 해당값에 대한 책임은 delegate class가 가지고 있으므로 그대로 가져올 수 있음

MaterialAlertDialog

마테리얼다이럴로그를 사용하면 깔끔하고 쉽게 적용가능

MaterialAlertDialogBuilder(requireContext())
            .setTitle(getString(R.string.congratulations))
            .setMessage(getString(R.string.you_scored, viewModel.score))
            .setCancelable(false)
            .setNegativeButton(getString(R.string.exit)) { _, _ ->
                exitGame()
            }
            .setPositiveButton(getString(R.string.play_again)) { _, _ ->
                restartGame()
            }
            .show()
